C + +, como usted ya sabe, es un lenguaje de alto nivel que se considera principalmente un superconjunto del lenguaje C, lo que añade muchas características, tales como la programación orientada a objetos y plantillas, pero lo que es CLI?

CLI es sinónimo de Common Language Infrastructure. Se explica a fondo en toda la web, pero en pocas palabras: Es una especificación abierta que describe el código ejecutable y entorno de ejecución que permite que múltiples lenguajes de alto nivel para su uso en distintas plataformas informáticas sin ser reescrito para arquitecturas específicas. [2]

Ahora, C + + / CLI es el medio para programar .NET en C + +, de manera similar como C # o VB.NET se utilizan.

C + + / CLI es un conjunto de extensiones a ISO C++ que proporciona una muy completa "vinculante" de C++ de Microsoft CLI (Common Language Infrastructure). Se ha normalizado por ECMA (ECMA-372). Me alegra que tiene todas las características del CLI estén fácilmente accesible de C++ por eso C + + / CLI es un lenguaje mucho mejor que su predecesor "C++ administrado". Sin embargo, no me gusta que C + + / CLI logra sus objetivos esencialmente aumentando C++ con una característica de la lengua separada para cada función del CLI (interfaces, propiedades, genéricos, punteros, herencia, enumeraciones y mucho, mucho más). Se trata de una importante fuente de confusión. La riqueza de las nuevas instalaciones de lenguaje en C + + / CLI en comparación con el estándar de ISO C++ tienta a los programadores escribir código no portable (a menudo invisible) se convierten en íntimamente ligado al Microsoft Windows.

El CLI ofrece un conjunto de interfaces (a sistemas) que son muy diferentes de las interfaces tradicionales a aplicaciones y servicios del sistema operativo. En particular, estas interfaces tienen semántica que no se puede expresar completamente o convenientemente en lenguajes de programación convencionales. Una manera de describir la CLI es como una "plataforma" (parcial) o "máquina virtual". Consta de un amplio conjunto de características de lenguaje (herencia, métodos, construcciones de bucle, mecanismos de devolución de llamada, etc.), apoyar un amplio conjunto de bibliotecas de la Fundación (la BCL), además de un elaborado sistema de metadatos. La CLI se describe a veces como "lenguaje neutro". Sin embargo, un lenguaje que no acepta que un gran subconjunto de estas instalaciones no puede utilizar instalaciones de .net incluso básicas (o futuras instalaciones de Microsoft Windows, suponiendo que no cambian los planes de Microsoft) y un lenguaje que no puede expresar todas estas características no pueden utilizarse para la aplicación de recursos destinado a ser utilizable por otros idiomas. Así, CLI es "lenguaje neutro" sólo en el sentido que cada lengua debe ser compatible con todas las características de la CLI para ser "primera clase" en .net.

 

Yo prefiero un enlace que unas primitivas, expresable como llamadas a funciones simples y estructuras de datos simples en cualquier idioma, posiblemente encapsulado en bibliotecas específicas de la lengua. Para la CLI, en el mejor esto puede hacerse para los consumidores de servicios CLI solamente. Un lenguaje que se utiliza para producir módulos CLI debe ser capaz de expresar todas las instalaciones de la CLI, incluidos los metadatos. Sólo un lenguaje que puede hacer puede considerarse un sistema de lenguaje en .net. Así, el equipo de Microsoft C++ llegó a la conclusión de que sólo construir-en idioma instalaciones son aceptables para sus clientes. Su diseño refleja una visión que no acepta absolutamente ninguna restricción en qué parte del CLI puede expresarse en C++ con C + + / extensiones CLI, absolutamente ningún detalle en comparación con otros idiomas utilizando instalaciones CLI y absolutamente sin gastos generales en comparación con otros idiomas. Tienen por objeto preservar C++ como los sistemas dominantes, lenguaje de programación para Windows.

Como siempre, un énfasis pesado en portabilidad y recomendar personas a aplicaciones de diseño para que el acceso a servicios específicos del sistema son a través de interfaces bien definidas en ISO C++ (por ejemplo, no usar C + + / CLI directamente). En Windows, esto a veces será inconveniente en comparación con el uso de C + + / instalaciones CLI directamente, sino de la única manera de obtener portabilidad y un grado de independencia del proveedor. Obviamente, este enfoque de la longitud de los brazos para CLI no puede ser mantenido si el propósito de una porción de código es proporcionar una interfaz CLI para ser consumido por otro código. Tenga en cuenta que yo reconozco la necesidad de extensiones específicas de sistema y que Microsoft no es el único proveedor de C++ con estas extensiones, yo apenas fuertemente prefieren tratar con tales extensiones a través de una "interfaz fina" especificado en ISO C++ estándar.

Cómo tratar con extensiones específicas de sistema es inherentemente una pregunta difícil. El equipo de Microsoft C++, especialmente la hierba Sutter, ha mantenido un diálogo activo con (otros) miembros del Comité de estándares ISO C++ para que la relación entre ISO C++ y su superset C + + / CLI eventualmente se elaboró. Tenemos un largo historial de conjunto constructivo trabajo en el Comité ISO C++. Además, para minimizar la confusión entre ISO C++ y C + + / extensiones CLI, Microsoft ahora está revisando la documentación de Visual C++ para intentar distinguir claramente C + + / CLI desde ISO C++ (llano no cualificado C++ significa ISO C++). Espero que otros seguirán que llevan.

Sobre la cuestión difícil y controversial de lo que debe ser llamado las CLI enlace/extensiones a C++, prefiero C + + / CLI como una abreviatura para "Las extensiones de la CLI para ISO C++". Mantener C++ como parte del nombre recuerda a la gente cuál es la lengua de la base y la voluntad ayudan a mantener C++ un subconjunto adecuado de C++ con el C + + / extensiones CLI. Los problemas de compatibilidad de C/C++ demuestran lo importante que es mantener esa propiedad de subconjunto.
